# Stocksie — Testing

> **Audience:** a contributor about to add a feature or fix a bug, who needs to
> know how to run the tests and where to add new ones. For the invariants the
> suite enforces see [SECURITY.md](SECURITY.md); for the account/state model see
> [ACCOUNTS.md](ACCOUNTS.md).

This is the polished output of the internal testing plan in
[`../plan/08_testing.md`](../plan/08_testing.md). Where this doc and the plan
disagree, the **code** is the source of truth — the test names below are the
**actual** names from the 75-test suite (enumerated via `cargo test -- --list`),
not the plan's aspirational list. A handful of plan-named tests were consolidated
during implementation; the matrix uses the real names.

## 1. Testing pyramid

```/dev/null/pyramid.txt#L1-11
                  ┌─────────────────────────┐
                  │   Cluster smoke tests   │  ← devnet/testnet, post-MVP
                  └─────────────────────────┘
                ┌───────────────────────────────┐
                │   Integration (Surfpool)      │  ← mainnet-fork, post-MVP
                └───────────────────────────────┘
              ┌─────────────────────────────────────┐
              │   LiteSVM (in-process Rust VM)      │  ← THE MVP LAYER
              └─────────────────────────────────────┘
            ┌──────────────────────────────────────────┐
            │   Pure unit tests (state machines, types)│  ← no VM, instant
            └──────────────────────────────────────────┘
```

The MVP lives in the bottom two layers:

- **Pure unit tests** (`#[cfg(test)]` blocks inside `src/`) verify domain logic
  that doesn't need accounts: the `Status` state machine, `Role::can_*()`
  helpers, `Household::next_request_id`, `Member::add_reward`, the
  `PurchaseRequest::transition_*` guards. They run in microseconds with no
  cluster and no compiled `.so`.
- **LiteSVM tests** (`programs/stocksie/tests/*.rs`) verify the full instruction
  surface against the real compiled `stocksie.so`: account creation, PDA
  derivation, constraint failures, lamport moves, event emission, and balance
  deltas.

Surfpool and cluster tests are deferred — they're valuable for mainnet-fork
realism but unnecessary to prove the MVP correct.

## 2. How to run the suite

The `.so` is `include_bytes!`-ed by the harness at compile time, so integration
tests require a successful `anchor build` first. Pure unit tests do not.

```/dev/null/run-tests.sh#L1-19
# Full suite (unit + integration). Requires target/deploy/stocksie.so.
anchor build
cargo test -p stocksie

# Pure unit tests only — no anchor build needed, runs in microseconds.
cargo test -p stocksie --lib

# LiteSVM integration tests only.
cargo test -p stocksie --tests

# One integration test file.
cargo test -p stocksie --test test_lifecycle

# One specific test (by name; matches across all targets).
cargo test -p stocksie double_reimburse

# Lint + format gates (must be clean before commit).
cargo clippy -p stocksie --all-targets -- -D warnings
cargo fmt --all -- --check
```

## 3. The LiteSVM harness

Every integration test shares one harness, extracted into
[`tests/helpers/mod.rs`](../programs/stocksie/tests/helpers/mod.rs). Test bodies
stay declarative — the harness is the single place that knows how to talk to
LiteSVM.

### Setup

A fresh LiteSVM with the compiled program loaded and a 100 SOL payer:

```programs/stocksie/tests/helpers/mod.rs#L105-119
pub fn setup_svm() -> (LiteSVM, Keypair) {
    let mut svm = LiteSVM::new();
    let program_bytes: &[u8] = include_bytes!("../../../../target/deploy/stocksie.so");
    svm.add_program(stocksie::id(), program_bytes)
        .expect("failed to load stocksie.so into LiteSVM");
    let payer = Keypair::new();
    svm.airdrop(&payer.pubkey(), 100_000_000_000) // 100 SOL
        .expect("airdrop of payer funds failed");
    (svm, payer)
}
```

### PDA derivation (mirrors `constants.rs` seeds exactly)

```programs/stocksie/tests/helpers/mod.rs#L121-143
pub fn derive_household(owner: &Pubkey) -> (Pubkey, u8) {
    Pubkey::find_program_address(&[HOUSEHOLD_SEED, owner.as_ref()], &stocksie::id())
}

pub fn derive_member(household: &Pubkey, wallet: &Pubkey) -> (Pubkey, u8) {
    Pubkey::find_program_address(
        &[MEMBER_SEED, household.as_ref(), wallet.as_ref()],
        &stocksie::id(),
    )
}

pub fn derive_request(household: &Pubkey, request_id: u64) -> (Pubkey, u8) {
    Pubkey::find_program_address(
        &[PURCHASE_SEED, household.as_ref(), &request_id.to_le_bytes()],
        &stocksie::id(),
    )
}
```

### Build + send an instruction

`build_ix` composes Anchor's generated `accounts::*` and `instruction::*` structs
into the canonical `discriminator(8) ++ borsh(args)` payload — the same bytes a
real wallet submits:

```programs/stocksie/tests/helpers/mod.rs#L165-178
pub fn build_ix<A, I>(accounts: &A, args: &I) -> Instruction
where
    A: ToAccountMetas,
    I: InstructionData,
{
    Instruction::new_with_bytes(
        stocksie::id(),
        &args.data(),
        &accounts.to_account_metas(None),
    )
}
```

### Assertions

| Helper                                | Returns             | Used to                                |
| ------------------------------------- | ------------------- | -------------------------------------- |
| `send(svm, payer, ix)`                | `TransactionResult` | execute an instruction                 |
| `account_of::<T>(svm, key)`           | `Option<T>`         | deserialize + read account state       |
| `balance_of(svm, key)`                | `u64`               | read lamports (vault, buyer, caller)   |
| `emitted_events_of::<E>(result)`      | `Vec<E>`            | decode + shape-check events            |
| `assert_error_code(failed, expected)` | —                   | assert a failed tx's `StocksieError`   |
| `rent_lamports(svm, data_len)`        | `u64`               | rent-exempt math for refund assertions |

Higher-level flow helpers compose these into reusable lifecycle stages:
`add_member`, `setup_two_member_household`, `reach_restocked`,
`reach_reimbursed`. A test that needs a request "ready to reimburse" calls
`reach_restocked(...)` instead of replaying create → approve → confirm by hand.

## 5. Negative-test pattern

Every permission/security test follows the same three-step shape:

1. **Positive control** — establish the happy path works (e.g. a Parent approves a
   request and the transition succeeds).
2. **Mutate one variable** — flip exactly one thing (the caller's role, the
   signer, the lifecycle state, the seed) and leave everything else identical.
3. **Assert the specific error variant** — via `assert_error_code`, which
   pattern-matches `TransactionError::InstructionError(_, InstructionError::Custom(n))`
   against the `StocksieError` discriminant.

This keeps negative tests surgical: a failure points at exactly the rule that
broke, not at a tangled setup. Example shape (a `Child` cannot approve):

The verbatim test lives at [`tests/test_permissions.rs`](../programs/stocksie/tests/test_permissions.rs) (`fn child_cannot_approve` at L147, through L176). Abridged view showing the mutate-one-variable shape (the full test also asserts the positive control first via a Parent approver):

```/dev/null/child-cannot-approve.rs#L1-30
fn child_cannot_approve() {
    let (mut svm, owner) = setup_svm();
    let (parent, child, household_pda, owner_member_pda, parent_member_pda, child_member_pda) =
        setup_three_member_household(&mut svm, &owner);

    // Positive control (elided): the Parent can approve this request.

    // Mutate one variable: the caller is now the Child (active member, role Child).
    let approve_ix = build_ix(
        &stocksie::accounts::ApprovePurchaseRequest {
            household: household_pda,
            caller_member: child_member_pda,
            request: request_pda,
            caller: child.pubkey(),
        },
        &stocksie::instruction::ApprovePurchaseRequest {},
    );

    // Assert the specific error variant — not just "it failed".
    let result = send(&mut svm, &child, approve_ix);
    assert!(result.is_err(), "Child approve should have failed");
    assert_error_code(result.unwrap_err(), StocksieError::UnauthorizedRole);
}
```

## 6. Event-assertion pattern

Anchor emits events as `Program data: <base64>` log lines. `emitted_events_of::<E>`
decodes those lines and deserializes them into the typed `#[event]` struct, so a
test asserts both **count** and **shape**:

```/dev/null/event-assert.rs#L1-12
use stocksie::events::{HouseholdCreated, MemberAdded};

let result = send(&mut svm, &payer, init_ix);
assert!(result.is_ok());

let created: Vec<HouseholdCreated> = emitted_events_of(&result);
assert_eq!(created.len(), 1);
assert_eq!(created[0].owner, owner.pubkey());
assert_eq!(created[0].name_hash, [1u8; 32]);

let added: Vec<MemberAdded> = emitted_events_of(&result);
assert_eq!(added.len(), 1);
assert_eq!(added[0].role, Role::Owner);
```

The invariant tests in [`test_invariants.rs`](../programs/stocksie/tests/test_invariants.rs)
push this further: they replay the **entire** event stream for a household and
assert cross-event invariants (treasury reconciliation, reward-sum equality,
membership-set consistency, no self-approval, slot monotonicity).

## 8. Coverage targets

The suite is held to these targets; a new instruction or invariant that isn't
covered is a gap to close before commit.

- **Every instruction has a positive and a negative test.** Each of the 14
  instructions has at least one happy-path exercise and at least one
  permission/argument-rejection test.
- **Every `StocksieError` variant is asserted.** Each variant is raised by at
  least one negative test (see [INSTRUCTIONS.md](INSTRUCTIONS.md) §6).
- **Every forbidden state transition is covered.** See the forbidden-transitions
  table in [ACCOUNTS.md](ACCOUNTS.md) §5 — each row maps to a named test.
- **Every security defense is verified** by a named test or explicitly marked
  "code review" in [SECURITY.md](SECURITY.md).
- **The privacy invariant is machine-checked.** `no_string_fields_on_chain`
  grep-asserts no `String` field exists on any `#[account]` or `#[event]` struct,
  and `scanner_sees_all_on_chain_structs` asserts the scanner reaches every
  struct.
- **The space budget is locked.** `test_space_budget` pins `INIT_SPACE` for all
  three accounts so a field addition is caught before it silently drifts rent.

## 9. Where to add a new test

```/dev/null/where-to-add.txt#L1-12
Is the behavior pure domain logic
(state machine, predicate, arithmetic)?
  │
  ├── yes ──▶ inline #[cfg(test)] mod tests in the
  │           source file that owns the logic.
  │           (cargo test -p stocksie --lib)
  │
  └── no (needs accounts / CPI / events / balances)
            │
            ├── is it a new invariant spanning
            │   multiple events or accounts?
            │     └── add a row to docs/TESTING.md §4.2 first,
            │        then write it in tests/test_invariants.rs.
            │
            └── otherwise ──▶ tests/test_<group>.rs,
                             using the helpers/mod.rs harness.
                             (cargo test -p stocksie --test test_<group>)
```

Rules of thumb:

- **Pure logic → inline unit test.** State-machine guards, role predicates, and
  arithmetic helpers belong in `#[cfg(test)] mod tests` inside the source file.
  They run without `anchor build` and give the tightest feedback loop.
- **Instruction behavior → `tests/test_<group>.rs`.** Anything that needs PDAs,
  signers, lamport moves, or events goes in the matching integration file
  (`test_household`, `test_lifecycle`, `test_permissions`, `test_reimburse`,
  `test_rewards`). Reuse `reach_restocked` / `reach_reimbursed` rather than
  replaying the lifecycle by hand.
- **Cross-cutting invariant → `tests/test_invariants.rs`.** If the assertion
  spans the whole event stream or multiple accounts (treasury reconciliation,
  reward-sum equality, monotonicity), add it here and add a row to §4.2.
- **Update this doc's matrix when you add a test.** Every shipped test should
  appear in §4 so the matrix stays a complete index of what the suite guarantees.
